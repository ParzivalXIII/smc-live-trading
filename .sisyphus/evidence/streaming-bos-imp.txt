Architecture
------------

Before (batch BOS):
  Phase 1: Candle → Swing Engine → (store swings)
  Phase 2: bos_choch(swings) → BOS retroactively stamped

After (streaming BOS):
  Phase 1: Candle → Swing Engine → Structure Engine → Events (real-time)
  Phase 2: bos_choch(swings) → still runs for per_candle_report (parallel)


Two-Stage StructureEngine
--------------------------

StructureEngine (smartmoneyconcepts/structures.py)
├── update(swing: SwingConfirmed) → list[StructureEvent]
│   └── Stage 1: 4-swing pattern detected → emit PROVISIONAL BOS/CHOCH
│       - Bullish BOS:  [-1, 1, -1, 1] with ascending level ordering
│       - Bearish BOS:  [1, -1, 1, -1] with descending level ordering
│       - Bullish CHOCH: same as BOS but different level ordering criterion
│       - Bearish CHOCH: same as BOS but different level ordering criterion
│       - swing_index = _swings[-3].index (S1, matches batch last_positions[-2])
│       - level = _swings[-3].level (S1's level, matches batch levels[-3])
│
├── check_confirmations(index, high, low) → list[StructureEvent]
│   └── Stage 2: Called EVERY bar
│       - Bullish: high > level → CONFIRMED
│       - Bearish: low < level → CONFIRMED
│       - > N bars with no break → CANCELLED (configurable window, default 10)
│
└── Dedup via _emitted_keys set: (type, direction, swing_index) prevents
    double-emission of the same pattern

SwingConfirmed (dataclass):
  index: int, direction: int (1/-1), level: float, timestamp: pd.Timestamp

StructureEvent (dataclass):
  event_type: str ("BOS" / "CHOCH")
  direction: int (1/-1)
  level: float
  swing_index: int
  status: str ("provisional" / "confirmed" / "cancelled")
  confirmed_at_index: int | None
  cancelled_at_index: int | None
  timestamp: pd.Timestamp | None

for i in range(len(data)):

        new_events = structure_engine.update(swing)     # Stage 1
        all_events.extend(new_events)

    # Every bar: check for break confirmations
    status_changes = structure_engine.check_confirmations(i, row["high"], row["low"])  # Stage 2
    all_events.extend(e for e in status_changes if e not in new_events)  # Dedup

    # Call strategy with events
    strategy_callback.update(i, row, result, simulator, structure_events=all_events)

Part 2: Streaming vs Batch Cross-Validation
---------------------------------------------

EURUSD-15M comparison:

  Batch BOS events:         546
  Streaming confirmed:      513
  Match rate:               94.0%  (well above 80% threshold)
  Cancelled (window expiry):  68
  Avg confirmation delay:   0.45 bars  (same-bar or next-bar mostly)

  Explanation of remaining 6% miss: structural timing gap.
  Batch scans for break from S2+2 onward; streaming only starts at S3 (4th swing).
  Any break in [S2+2, S3-1] is permanently missed by streaming. This is the
  inherent cost of causal detection and is documented as a known limitation.

Key Design Decisions
---------------------

Decision                          Choice
────────                          ──────
BOS emission index                Two-stage: provisional on 4th swing, confirmed on break
Unbroken signals                  Cancelled after configurable window (default 10 bars)
Break detection                   high > level (bullish), low < level (bearish)
Break level                       levels[-3] = S1's level (matches batch)
Swing index for event             _swings[-3].index (matches batch last_positions[-2])
Strategy integration              structure_events list per bar, filtered by status=="confirmed"
Batch bos_choch()                 Still runs for per_candle_report (parallel, not replaced)
Default bos_confirmation_window   10 bars

Known Limitations
------------------

1. Timing Gap: Breaks in [S2+2, S3-1] are permanently missed by the streaming
   engine. Match rate against batch is structurally <100% and depends on average
   swing spacing. 94% observed on EURUSD-15M.

2. End-of-dataset Limbo: Provisional events emitted in the last
   bos_confirmation_window bars will never resolve (remain "provisional").
   Strategy ignores them (filters on "confirmed"). ~5-15 per dataset.

3. close_break Asymmetry: Streaming checks high/low for break detection (always).
   Batch can optionally use close_break=True (close > level). This means streaming
   confirms more events than batch when close_break=True — high > level is a
   superset of close > level.
